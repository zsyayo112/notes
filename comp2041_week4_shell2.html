🐛Tutorial Q3 — 波浪号在双引号里失效

Shruti 的符号链接脚本出 bug，原因是 "~/friends/$image_file" 里的 ~ 不会被展开：

broken symlinks
# ❌ 错误版本 — 波浪号在双引号内不展开
for image_file in $(ls ~/friends); do
ln -s "~/friends/$image_file" .
done

# 链接目标是字面量字符串 "~/friends/xxx"，不是真实路径
ls -la photo.jpg
photo.jpg -> ~/friends/photo.jpg ← 这个路径根本不存在！
✅ 三种修复方案
正确写法
# 方案 1：用 $HOME 变量（在双引号里会展开）
ln -s "$HOME/friends/$image_file" .

# 方案 2：把 ~ 放在引号外面
ln -s ~"/friends/$image_file" .

# 方案 3：用 glob 代替 ls（更健壮，避免文件名含空格问题）
for image_file in ~/friends/*; do
ln -s "$image_file" .
done
⚠️ 额外陷阱：不要用 ls 遍历文件
for f in $(ls dir) 会在文件名含空格时出错。应直接用 glob：for f in dir/*

04 循环 (for / while)
🔄is_prime.sh + primes.sh — Tutorial Q9
is_prime.sh
#!/bin/dash
n
=$1
if [ "$n" -lt 2 ]; then
echo "$n is not prime"; exit 1
fi
i
=2
while [ "$i" -lt "$n" ]; do
if [ $(( n % i )) -eq 0 ]; then
echo "$n is not prime"; exit 1
fi
i=$(( i + 1 ))
done
echo "$n is prime"; exit 0

👁️recompile.sh — Tutorial Q10 & Q11
recompile.sh (stat 版本)
#!/bin/dash
cfile
=$1
last_mod
=0
while true; do
current_mod=$(stat -c '%Y' "$cfile")
if [ "$current_mod" -ne "$last_mod" ]; then
last_mod="$current_mod"
if gcc "$cfile" -o a.out; then
./a.out
fi
fi
sleep 1
done
recompile.sh (inotifywait 版本 — 更高效)
#!/bin/dash
cfile
=$1
while inotifywait -e close_write "$cfile"; do
if gcc "$cfile" -o a.out; then
./a.out
fi
done
💡 为什么用 inotifywait 更好？
stat 版本每秒轮询一次，持续消耗 CPU。inotifywait 让内核在文件发生变化时才唤醒进程，CPU 开销几乎为零。

06 管道、find 与 PATH 搜索
🛤️which.sh — Tutorial Q13

手动实现 which：在 PATH 的每个目录里用 ls 查找可执行文件：

which.sh
#!/bin/dash
program
=$1
found
=0
# 用 tr 把 PATH 里的冒号换成换行，逐行处理
echo "$PATH" | tr ':' '\n' | while IFS= read -r dir; do
if [ -x "${dir}/${program}" ]; then
ls -la "${dir}/${program}"
found=1
break
fi
done
if [ "$found" -eq 0 ]; then
echo "$program not found"
fi

./which.sh cat
-rwxr-xr-x 1 root root 43936 Sep 24 18:36 /bin/cat
./which.sh lost
lost not found
⚠️ 目录名含空格的处理
使用 IFS= read -r dir 而不是 for dir in $(echo $PATH | tr ':' ' ')，防止含空格的目录名被错误分割。

📋Shell 核心命令速查
命令作用关键选项
sed -i 's/old/new/g' file 原地替换文本 -i (in-place), g (全行替换)
find dir -type f 递归查找文件 -type f/d, -name "*.c"
stat -c '%Y' file 打印文件修改时间（秒） -c 格式字符串
inotifywait -e close_write f 等待文件被写入后退出 -e 事件类型
tr ':' '\n' 字符替换 / 分割 -d 删除, -s 压缩
grep -oE 'pattern' 只输出匹配部分 -o 只匹配部分, -E 扩展正则
sort | uniq -c | sort -rn 计频 + 按频次排序 -r 逆序, -n 数值

Q11
解释 inotifywait 比 stat 轮询方案更好的原因，并写出基于 inotifywait 的 recompile.sh 核心循环。
›
答案
#!/bin/dash
cfile
=$1
while inotifywait -e close_write "$cfile" 2>/dev/null; do
echo "=== File changed, recompiling... ==="
if gcc "$cfile" -o a.out; then
echo "=== Running: ==="
./a.out
fi
done
轮询 (stat) 的问题：每秒都要运行 stat 命令，即使文件没变，也消耗 CPU 和电池。

inotifywait 的优势：它向 Linux 内核注册对文件系统事件的监听（inotify API），进程会 阻塞休眠 直到文件真的发生变化，CPU 占用接近于零。这种机制叫做 事件驱动，比轮询高效得多。
较难
Q12
写 which.sh，接受一个程序名，在 $PATH 各目录里查找该程序并用 ls -la 显示详情。若找不到打印 NAME not found。要求能处理目录名含空格的情况。
›
答案
#!/bin/dash
program
=$1
found
=0

# 把 PATH 中的冒号替换为换行，用 read 安全逐行读取
echo "$PATH" | tr ':' '\n' | while IFS= read -r dir; do
target="${dir}/${program}"
if [ -x "$target" ] && [ -f "$target" ]; then
ls -la "$target"
exit 0 # 找到了，退出 while 的子 shell
fi
done || echo "$program not found"

./which.sh cat
-rwxr-xr-x 1 root root 43936 /bin/cat
./which.sh xyz123
xyz123 not found
关键点：
1. tr ':' '\n' 分割 PATH
2. IFS= read -r dir 安全读取（支持含空格目录名，不分割）
3. -x 检查可执行，-f 确认是文件而不是目录
4. 管道中的 exit 0 只退出子 shell（while 在管道里），所以用 || echo "not found" 在整个管道失败时打印找不到的信息。
较难
Q13
解释下面代码为什么 found 变量在循环后始终是 0，即使循环内确实赋值了 1：
found=0
echo "$PATH" | tr ':' '\n' | while read dir; do found=1; done
echo $found
›
答案
这是经典的 管道子 Shell 陷阱。

在 Shell 中，管道的每个部分都在 独立的子 Shell 中运行。因此 while read dir; do found=1; done 运行在子 Shell 里，对 found 的赋值只影响子 Shell 的变量，父 Shell 的 found 不变，依然是 0。

解决方案：
1. 用进程替换（bash）：while read dir; do found=1; done < <(echo "$PATH" | tr ':' '\n')
2. 用临时文件：echo "$PATH" | tr ':' '\n' > /tmp/dirs; while read dir; do found=1; done < /tmp/dirs
3. 在子 Shell 里用 exit 状态传递结果，然后用 $? 检查。
较难
Q14
下面的 for 循环有什么潜在 bug？如何修复？
for f in $(ls ~/photos); do echo "$f"; done
›
答案
存在 两个 bug：

Bug 1：单词分割 — $(ls) 的输出经过单词分割，含空格的文件名（如 my photo.jpg）会被拆成两个词 my 和 photo.jpg，循环会错误地把它们当成两个单独的文件名。

Bug 2：波浪号在 $() 外但结果仍然是字符串 — 这里 ~/photos 会展开，但 ls 的输出是文件名而不是路径，后续处理时会缺少完整路径。

✅ 修复：用 glob
# ✅ 正确：glob 不会做单词分割，且保留完整路径
for f in ~/photos/*; do
echo "$f"
done
